Adv. Competency – Accessible Design

May 2, 2026 · 3 min · Linda Jing

1. Identify a portion of the user interface that you designed.#

The portion of the interface I focused on is the course planning and requirement-checking area of the four-year planner. This includes the major selection dropdown, the eight-semester course grid, the Auto-Fill Remaining Plan button, the Check Requirements button, and the requirement cards below the planner.

This part of the interface is important because it is where students actually interact with the planner. They choose a major, add courses, check progress, and read whether requirements are completed or still missing.

2. How does your design adhere to one or more principles of accessible design?#

My design follows accessible design principles by making the planner more perceivable, operable, and understandable.

First, the interface is visually organized into clear sections. The major selector and action buttons are at the top, the semester plan is in the middle, and the requirement results are below. This helps users understand the workflow: choose a major, build a plan, then check requirements.

Second, the requirement cards use text labels instead of relying only on color. Even though completed and incomplete requirements are visually different, the interface also includes written information such as requirement names, course options, progress counts, and notes. This is important because users should not have to rely only on color to understand their progress.

Third, the buttons use clear action words such as Auto-Fill Remaining Plan, Check Requirements, and Reset Plan. These labels make the actions understandable for users who are unfamiliar with the tool.

Fourth, the semester layout uses repeated structure. Each semester card has the same format, with a semester title and course slots. This consistency reduces cognitive load because users do not have to relearn the interface for each semester.

3. How did you use validation tools, if at all, to ensure that your design was accessible?#

I did not use a full accessibility testing suite during the early design process, so this part of the project was mostly validated through manual checks. I checked whether the interface could be understood visually, whether buttons and labels were clear, and whether the requirement results could be interpreted from text rather than color alone.

For a stronger validation step, I would use browser accessibility tools such as Chrome Lighthouse or axe DevTools to check for issues like color contrast, missing labels, heading structure, and keyboard navigation. I would also manually test whether a user can tab through the major dropdown, buttons, and course controls in a logical order.

The main accessibility limitation of the current design is that it still needs more formal validation. In the future, I would improve the interface by adding stronger keyboard support, clearer focus states for course slots and expandable requirement cards.
