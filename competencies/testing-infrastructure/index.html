Testing Infrastructure

April 30, 2026 · 3 min · Linda Jing

1. Identify relevant components from your development infrastructure that you use to test your project.#

The relevant components from my development infrastructure are VS Code, the integrated terminal, npm scripts, Node’s built-in test runner, Vite, PostgreSQL, and the browser.

VS Code was my main IDE for writing and editing the test files. The integrated terminal let me run test commands directly from the project workspace. Node’s built-in test runner was used to execute the unit tests in the tests/ folder. npm scripts made those commands easier to run repeatedly. I also used Vite and the browser for manual frontend testing, and PostgreSQL/psql for checking whether the SQL-backed planner data could be loaded correctly.

2. How have you automated your tests so that they both take “1-click” to run and run automatically as part of build validation?#

I automated the tests by adding scripts to package.json:

"test": "node --test tests/*.test.js",
"coverage": "node --test --experimental-test-coverage tests/*.test.js"

This means the full test suite can be run with one command:

npm test

The coverage report can also be generated with one command:

npm run coverage

This is close to a “one-click” test workflow because I do not need to manually run each test file. The Node test runner automatically finds and runs the test files in the tests/ folder.

These tests are not yet connected to GitHub Actions, so they do not automatically run on every push yet. However, the project is structured so they could easily become part of build validation by running:

npm test
npm run build

A future GitHub Actions workflow could run both commands before accepting a pull request.

3. Describe a specific occurrence in which your testing infrastructure saved you and/or your team work.#

One specific issue involved the Computer Science math requirement. The planner correctly recognized that a math course above MAT-131 could satisfy the CS math elective requirement. However, this created a bug where the planner could choose a 200-level math course and count the requirement as completed, even if the student had not taken the prerequisite needed to register for that course.

The testing infrastructure helped because I could separate the problem into two smaller checks. The requirement unit tests checked whether courses like MAT-133 and STA-209 counted for the CS math elective. The prerequisite unit tests checked whether a course could actually be placed in a semester before its prerequisite was completed.

This saved debugging time because the team did not have to guess whether the problem came from the requirement checker, the prerequisite logic, the frontend UI, or the SQL data. By running npm test, I could verify the small logic rules directly before checking the full planner manually in the browser.
