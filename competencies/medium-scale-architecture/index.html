Medium-Scale Architecture

April 26, 2026 · 4 min · Linda Jing

For medium-scale architecture, I would use the database-side architecture of the project as my example. I designed and wrote the core SQL schema structure in 00_core_schema.sql, and I also wrote the major seed files that follow this structure. Instead of storing every major in a completely different format, the project uses one shared relational database architecture. Each major file inserts its courses, requirement blocks, course options, and special group logic into the same set of tables.

This architecture is visible in 00_core_schema.sql, where the project defines shared tables such as courses, majors, requirement_blocks, block_course_options, block_or_children, subfields, and course_subfields. Then individual major files, such as computerscience.sql and anthropology.sql, use those tables to represent their own degree requirements.

Refer to the whole sql folder.

1. What architectural pattern does your program employ?#

My part of the program uses a normalized relational database architecture. The main idea is that the planner stores academic information as connected tables instead of as separate hard-coded objects for each major. Courses, majors, requirements, requirement options, and subfields are separated into different tables, and the relationships between them are represented through primary keys, foreign keys, and join tables.

For example, the courses table stores course information once, while requirement_blocks stores the general requirement units. The block_course_options table connects a requirement block to the courses that can satisfy it. This follows a normalized relational pattern because course information is not repeated inside every requirement. Instead, requirements refer back to shared course records.

This architecture also supports different relationship types. A major has many requirement blocks. A requirement block can have many course options. Some requirements can branch into multiple valid paths through block_or_children. Some courses can also belong to subfields through course_subfields. This makes the SQL structure more flexible than a flat table or a separate file format for every major.

2. What components result from this pattern in your program?#

This pattern creates several main database components.

The first component is the course catalog layer. This is represented by the courses table in 00_core_schema.sql. It stores general course information such as department, course number, course code, title, credits, level, and active status. This lets different majors reuse the same course records instead of duplicating them.

The second component is the major layer. This is represented by the majors table. Each major seed file inserts a major into this table, such as CSC for Computer Science or ANTH for Anthropology. This gives every requirement block a major to belong to.

The third component is the requirement layer. This is represented by the requirement_blocks table. Each row is one degree requirement, such as a required course, a choose-one category, a choose-n category, a credit requirement, or a custom rule. The rule_type column controls what kind of requirement it is.

The fourth component is the course-option layer. This is represented by block_course_options. It connects requirement blocks to the courses that can satisfy them. For example, a choose-one requirement can point to several possible courses without duplicating the requirement itself.

The fifth component is the branching layer. This is represented by block_or_children. This table lets one parent requirement represent an either/or structure by pointing to child requirement blocks. This is useful for requirements where students can complete one of multiple valid paths.

The sixth component is the subfield layer. This is represented by subfields and course_subfields. This part is useful for majors like Anthropology, where courses may count toward different subfields such as archaeology, biological anthropology, cultural anthropology, or linguistic anthropology.

3. What technologies and/or libraries make up each of the components?#

The architecture is built with SQL using a PostgreSQL-style relational design. The core schema is written in 00_core_schema.sql, and the data for each major is written in separate .sql seed files inside the sql/majors/ folder.

The main technology is SQL itself, especially relational tables, primary keys, foreign keys, unique constraints, check constraints, and join tables. For example, block_course_options is a join table between requirement blocks and courses. course_subfields is another join table between courses and subfields.

The major files also use repeatable SQL seed patterns such as INSERT INTO … SELECT … JOIN … and ON CONFLICT DO NOTHING. This allows the seed files to be rerun without duplicating existing rows. That is important because the database can be rebuilt or updated while keeping the same architecture.

Overall, this architecture makes the project more scalable. To add a new major, I do not need to invent a new data format. I can add a new major seed file that inserts courses and requirement blocks into the existing tables. The structure of the program stays consistent even when the academic requirements are different across majors.
