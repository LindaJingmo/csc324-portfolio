Medium-Scale Abstraction

April 26, 2026 · 3 min · Linda Jing

I designed and wrote the requirement_blocks table structure, the block_course_options table, and the block_or_children table in 00_core_schema.sql myself. It aims to unify all requirement types under a single table for each major file with different requirements for graduation. I also wrote the major seed files that consume this abstraction, including computerscience.sql and the structure used across all other major files. Generative AI was used to tell me what fields needed for schema design. But the structure I changed was based on how to model real Grinnell degree requirements without concentration and major. Even though we did not end up using concentration in our final product, I still consider this as a step and attempt for me to do abstraction.

1. What functionality does your code sample abstract away?#

The requirement_blocks table abstracts away the different degree requirement logic that exists across all majors in the planner. At Grinnell, degree requirements come in many forms: some are single mandatory courses, some ask a student to pick one course from a list, some require a minimum number of credits from a pool, and some branch into two entirely different valid paths. Without this abstraction, each major would need its own bespoke set of tables or hard-coded logic to represent these rules, and any code that checks whether a student has satisfied their degree. The abstraction hides all of that variation behind a single, uniform interface that every major in the system writes to and every audit query reads from.

2. How does it mechanically achieve abstraction?#

The abstraction is implemented as a relational table (the rule_type field in the first function in screenshot) that uses a rule_type discriminator column to distinguish between six different kinds of requirement logic: must_take, choose_one, choose_n, choose_credits, or_group, and custom. Callers inserting a requirement for all majors always write to the same requirement_blocks table using the same columns. The rule_type value is what signals to audit logic how that particular block should be evaluated. For branching requirements (like Either (A) two 300-level ANT courses OR (B) one 300-level ANT + ANT 499 (thesis)), the companion block_or_children table allow one or_group block to point to two child blocks, modeling an either/or path without requiring any schema changes. The result is that all twelve majors in the project can express their requirements through identical SQL patterns, even though their underlying rules are structurally quite different.

3. What purpose does this abstraction serve in your larger program?#

This abstraction is the foundation that makes the entire multi-major planner feasible as a single unified system. Every major file in the project — Computer Science, Economics, Art History, Biology, Chemistry, Chinese, Japanese, French, English, Anthropology, Biological Chemistry, and GWSS — writes its requirements into requirement_blocks using the exact same structure, which means any frontend audit logic or degree-progress query only needs to be written once. If this abstraction did not exist and each major instead stored its requirements in its own format, adding a new major would require not just a new data file but new audit code, new display logic, and new edge-case handling for every variation in requirement type — the project would scale very poorly. Because all requirement diversity is captured inside the rule_type column and the supporting option tables, the rest of the system remains stable and consistent regardless of how many majors or how many rule variations are added.
