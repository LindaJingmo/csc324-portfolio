May 2, 2026 · 4 min · Linda Jing

Instance 1: Resolving a Bug#

Category: Filing and/or resolving a bug

Evidence:

Identify the work you did in this instance.#

In this instance, I revised the prerequisite logic after code review feedback. The bug was that the planner could count or place a course before checking whether the student had completed the prerequisite. This was especially important for the Computer Science math requirement, because the planner could choose a 200-level math course that satisfied the math elective rule, even though the student might not actually be able to register for it yet. I updated the logic so course placement checks prerequisites before adding a course to the plan.

Describe the difficulties that necessitated communication with your peers in this instance.#

This required communication because the bug was not only a backend or frontend issue. It affected requirement checking, auto-fill behavior, and the user interface. A course could make a requirement look completed, but the overall plan could still be invalid because the prerequisite sequence was wrong. We had to discuss whether the issue came from the requirement checker, the prerequisite logic, or the way the frontend displayed completed requirements.

Did you encounter any difficulties in collaboration with the team during this instance? If so, how did you resolve those difficulties?#

The main difficulty was that different teammates were working on different layers of the project, so the same bug looked different depending on where we saw it. To resolve this, we separated the problem into smaller pieces: whether a course counts for a requirement, whether it can be placed in that semester, and whether the UI shows the correct status. After that, I revised the prerequisite logic based on the review feedback.

Instance 2: Filing a Pull Request of Linking database#

Category: Filing a pull request

Evidence:

Identify the work you did in this instance.#

In this instance, I filed a pull request to connect the React planner with SQL-backed data. My work focused on moving the planner away from only using JSON and toward loading course, major, and requirement information from the SQL-backed structure. This included coordinating the database data with the frontend so the planner could display majors, courses, and requirement cards correctly.

Describe the difficulties that necessitated communication with your peers in this instance.#

This required communication because Drilon and Nahom were working mainly on the frontend and UI design, while I was working on the SQL/database side. The frontend needed a stable data shape to display courses and requirements, but the SQL connection was still being developed. We had to coordinate what information the frontend expected and how SQL data would eventually replace the temporary JSON data.

Did you encounter any difficulties in collaboration with the team during this instance? If so, how did you resolve those difficulties?#

The main difficulty was that the frontend could not wait for the entire SQL connection to be finished. To avoid blocking the team, we used JSON as an intermediate data source so frontend development could continue. Once the SQL-backed data was ready, this PR helped integrate the SQL connection into the planner. Communication helped us make sure the frontend behavior still matched the new data source.

Instance 3: Filing a Pull Request of Collaborative Feature Integration#

Category: Filing a pull request

Evidence:

Identify the work you did in this instance.#

In this instance, I filed a pull request for the prerequisite logic feature. My work was to make the planner consider prerequisite relationships when deciding whether a course could be placed into a semester. This was related to auto-fill and manual course placement because the planner needed to avoid creating a schedule where advanced courses appeared before their required earlier courses.

Describe the difficulties that necessitated communication with your peers in this instance.#

This required communication because prerequisite logic changes how the frontend planner behaves. If the logic blocks a course from being placed, the UI needs to make sense to the user. I had to coordinate with teammates working on the frontend so that the course placement behavior would still feel understandable and not just silently fail.

Did you encounter any difficulties in collaboration with the team during this instance? If so, how did you resolve those difficulties?#

The main difficulty was deciding how strict the planner should be. If the planner is too permissive, it creates invalid schedules. If it is too strict without explanation, users may not understand why a course cannot be added. We resolved this by focusing first on the core logic: blocking courses when prerequisites were missing and allowing them when prerequisites were completed. Then the frontend could display or handle those placement results more clearly.
