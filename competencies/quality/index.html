Quality

May 2, 2026 · 3 min · Linda Jing

1. Overall evaluation of the codebase quality#

Overall, I would evaluate the codebase as a strong student project with a thoughtful architecture and a clear product goal, but not yet production-quality software. The project shows engineering quality in the way it separates frontend planning behavior, backend API routes, and SQL-backed academic data. It also shows care in modeling real degree requirements rather than hard-coding every major directly into the UI.

The strongest part of the codebase is the SQL/data model. The project uses shared tables such as courses, majors, requirement_blocks, block_course_options, and block_or_children to represent many different kinds of academic requirements. This makes the planner more scalable than a version where every major has its own special format. The React frontend can then consume requirement data in a more consistent way.

At the same time, the codebase still has limitations. Some requirement logic is still custom, some major data requires manual checking, and the project is not fully deployed with a cloud database. The test suite covers important requirement and prerequisite logic, but it does not cover the entire UI, API, or database behavior.

2. In what ways does the codebase exemplify engineering excellence?#

The codebase exemplifies engineering excellence first through its abstraction of academic requirements. Instead of representing each major as a completely different object, the SQL schema uses a common structure for requirement blocks. Different requirement types, such as required courses, choose-one groups, choose-n groups, credit requirements, and custom rules, can be represented through the same table structure. This reduces duplication and makes the system easier to extend.

Second, the project uses a layered architecture. The React frontend handles user interaction, the Node/Express API handles data access, and PostgreSQL stores structured course and requirement data. This separation makes the system easier to reason about than a single file containing all course data and all UI logic.

Third, the codebase uses repeatable setup scripts. The database can be rebuilt through sql/99_run_all.sql, and the app can be run with npm scripts such as npm run dev, npm run server, and npm test. This improves developer workflow because the project does not depend entirely on manual setup.

Fourth, the project includes verification work. Unit tests were added for requirement-checking logic and prerequisite placement logic. These tests do not cover the whole project, but they target important logic where correctness matters. Coverage tooling was also used to measure how much of the isolated requirement logic was exercised by the test suite.

Fifth, the product design is connected to real user needs. The planner addresses a concrete problem: students currently have to combine catalog pages, major requirements, advisor notes, and spreadsheets. The UI brings course planning and requirement progress into one place.

Sixth, the codebase shows attention to data correctness. Major requirement data is modeled in SQL rather than only hard-coded into React. This makes academic rules easier to inspect, update, and compare against the course catalog.

3. In what ways does the codebase fall short?#

The biggest weakness is that the project is not fully production-ready. The PostgreSQL database currently works best as a local development database. For the planner to work on any computer as a public website, the project would need a deployed backend and a cloud database, or a static JSON fallback.

Another weakness is that some requirement rules are still too complex for the current automated checker. Some custom rules, such as multi-path requirements or policy constraints, require manual review. This means the planner can help students, but it should not be treated as an official degree audit system.
