Adv. Competency – Security

April 26, 2026 · 2 min · Linda Jing

1. What is the “surface area” of your project?#

Our project is a Grinnell four-year course planner. Students can select a major, add courses into an eight-semester plan, auto-fill remaining requirements, and check whether their plan satisfies major requirements. The system uses a React frontend, a Node/Express API, and a PostgreSQL database during local development.

The surface area includes the places where users or developers interact with the system:

  • React frontend: major dropdown, course slots, auto-fill button, check requirements button
  • Course input/dropdown: users can add or change courses
  • API routes: /api/courses, /api/majors, /api/majors/:code/requirements
  • PostgreSQL database: stores courses, majors, requirements, prerequisites, and course options
  • .env file: stores local database settings
  • SQL seed files: define and populate academic data
  • Local development commands: npm run dev, npm run server, psql -f sql/99_run_all.sql

2. What are the possible threats against your project?#

Threat 1: SQL Injection

An outside user sends a malicious major code or API request to attack the PostgreSQL database to read, corrupt, or delete planner data.

  • Actor: outside user
  • Action: sends malicious input through API routes
  • Attack: SQL injection
  • Asset: PostgreSQL course and requirement database
  • Objective: access or damage planner data

Threat 2: Leaked Database Credentials

A developer accidentally commits .env to GitHub, exposing database credentials and allowing others to connect to the database.

  • Actor: developer or outside GitHub viewer
  • Action: exposes or reads .env
  • Attack: credential leak
  • Asset: database username, password, database name
  • Objective: gain unauthorized database access

Threat 3: Incorrect Academic Requirement Data

A developer accidentally enters incorrect SQL seed data, causing the planner to tell students that a requirement is completed when it is not.

  • Actor: developer
  • Action: changes requirement seed files incorrectly
  • Attack: data integrity failure
  • Asset: academic requirement correctness
  • Effect: students receive misleading course-planning advice

3. How will you handle the threats in your project?#

SQL Injection is high priority if the app is deployed. The mitigation is to use parameterized SQL queries and validate major codes instead of directly building SQL strings from user input.

Leaked database credentials mitigation is to keep .env in .gitignore, only commit .env.example, and avoid putting real database credentials in GitHub.

Incorrect academic requirement data is important because the main value of the planner depends on correctness. The mitigation is to review SQL seed files against the Grinnell catalog, rerun the database seed script, and manually test major fulfillment in the UI.
