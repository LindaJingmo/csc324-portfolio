Tools For Artifact

April 26, 2026 · 3 min · Linda Jing

Tools Used#

  • VS Code — main IDE used to write and edit SQL, React, and configuration files.
  • VS Code Live Share — collaboration tool so teammates could inspect files and debug together in the same workspace.
  • Git/GitHub — version control to track changes, share code, and prepare the project for submission.
  • Integrated Terminal — used to run development commands without leaving VS Code.
  • npm scripts — used to start repeatable project commands such as the frontend and backend.
  • Vite — local React development server used to run the frontend at localhost:5173.
  • Node/Express — backend runtime and API layer used to connect the React frontend to SQL data.
  • PostgreSQL / psql — database and command-line tool used to create, seed, and validate the SQL-backed planner database.
  • Browser / localhost — used to manually test the planner UI and check whether the frontend, backend, and database were connected correctly.

1. What development tools did you use through the process to help write code and automate the build/deployment process?#

For development infrastructure, I used VS Code, VS Code Live Share, Git/GitHub, the integrated terminal, npm scripts, Vite, Node/Express, and PostgreSQL. VS Code was my main IDE for editing SQL, React, and configuration files. VS Code Live Share helped with collaboration because teammates could inspect the same workspace and debug files together. Git/GitHub helped track project changes and share code with the team.

For running and validating the project locally, I used the terminal and npm scripts. The frontend could be started with npm run dev, the backend API with npm run server, and the database could be rebuilt with:

psql -h localhost -d planner -f sql/99_run_all.sql

These commands created a repeatable local deployment process because I could start each part of the system in a predictable way.

2. What is one specific problem that you encountered while writing code?#

One specific problem I encountered was when the React planner showed that it could not load planner data from SQL. At first, it was hard to tell whether the problem came from the React frontend, the backend API, or the PostgreSQL database. Since the project had multiple layers running at the same time, one missing process could make the whole app appear broken.

For example, if the backend API was not running, the Vite frontend showed proxy errors like ECONNREFUSED. If the backend was running but the database settings were wrong, the API returned a database connection error. This made debugging confusing at first because the visible error appeared in the browser, but the real cause could be in a different part of the stack.

3. How did your tools help or hinder your ability to address that problem?#

The tools helped because they let me debug the project layer by layer. The browser showed whether the React UI loaded. The Vite terminal showed frontend proxy errors. The backend terminal showed whether the API server started correctly. PostgreSQL and psql let me rebuild the database and confirm that the SQL files worked. Testing localhost:3001/api/health helped me check the backend separately from the frontend.

VS Code Live Share also helped because a teammate could inspect the same SQL or configuration files while I ran the project commands. This made it easier to compare the browser error, terminal output, and database settings at the same time.

The tools also hindered debugging a little because there were many moving parts: Vite on localhost:5173, the API server on localhost:3001, and PostgreSQL on port 5432. If one part was not running, the error could look like a different problem. After this issue, I used a clearer debugging order: first check the frontend, then the API health route, then .env, then PostgreSQL, and finally the SQL seed files.
