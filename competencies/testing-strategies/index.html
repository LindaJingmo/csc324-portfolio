Testing Strategies

April 26, 2026 · 2 min · Linda Jing

The component I was responsible for testing was the requirement-checking logic for the four-year planner. This logic decides whether the courses in a student’s plan satisfy a major requirement. My artifact for this competency is tests/requirementLogic.test.js.

2. According to code coverage tools, what aspects of this component are covered with tests?#

I did not configure a formal percentage-based coverage tool such as Istanbul or Vitest coverage. However, my unit tests cover important branches of the requirement-checking logic, including:

  • Required-course checks
  • Missing required courses
  • Choose-one rules
  • Choose-n rules
  • Credit-counting rules
  • The custom Computer Science math elective rule

3. What kinds of tests are they?#

They are unit tests. They test one isolated logic function without needing to start the React frontend, backend API, or PostgreSQL database. The tests create sample courses and requirement blocks, then check whether the requirement returns true or false for different planned course lists.

4. For code that is not covered by tests, why did you not cover them?#

The code not covered includes the full React UI, the backend/database connection, and the exact auto-filled four-year schedule. I did not test the exact full schedule because there can be multiple valid four-year plans, so one fixed expected output would be too brittle. I also did not cover the full UI because it depends on browser interaction and visual behavior.
